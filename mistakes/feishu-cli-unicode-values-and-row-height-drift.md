# 飞书 CLI 中文二维数组与导出行高漂移

## 基本信息
- 日期：2026-07-15
- 分类：飞书电子表格 / Windows CLI / 导出验证
- 任务：补建三冰与 Dark War 前期体验拆解页签

## 问题现象
- 旧 `sheets +write --values` 在 Windows `.cmd` 参数链中写入包含中文字符串的二维数组时，
  CLI 报 `--values invalid JSON, must be a 2D array`。
- 改用新版 `+cells-set --cells @file` 后，系统临时目录的绝对路径又被 CLI 安全策略拒绝；
  该接口只允许当前工作目录内的相对文件路径。
- Dark War 导出 xlsx 的标题和长文本行比目标换算值高约 3～4.4pt；
  内容未裁切，但原验证器使用对称 ±2pt 容差，导致合格的偏高行高被误判并触发事务清理。

## 原因分析
- `.cmd` 参数转发会破坏带引号的中文 JSON 字符串；纯数字二维数组可以解析，含字符串时稳定失败。
- 新版复合参数文件限制在当前目录内，系统 Temp 绝对路径不满足其路径安全规则。
- 飞书导出到 xlsx 时，字体最小高度与行高换算存在正向量化漂移；偏高少量不会造成文本不可见，
  与偏低导致裁切的风险不对称。

## 修复动作
- 单元格写入迁移到 `sheets +cells-set`，将普通值转换为 `{value: ...}` 单元格对象。
- `LarkRunner` 把中文 cells JSON 临时写到当前工作目录，使用 `@./<file>` 相对引用，
  无论成功或失败都删除临时文件。
- 行高验证改为非对称容差：允许实际值比目标低最多 2pt、比目标高最多 6pt；
  仍拒绝明显偏低、裁切风险或异常过高。
- 增加回归测试，覆盖中文文件参数、相对路径、临时文件清理、新版写入调用及行高正向漂移。
- 所有失败页签均由事务自动删除；三冰和 Dark War 最终版本重新创建并通过回读、合并和 xlsx 复核。

## 预防办法
- Windows 下禁止再把中文二维数组直接放进 `--values` 命令参数。
- 复合 JSON 优先使用当前目录内的临时文件和 `@file`，并在 `finally` 中清理。
- 样式验证优先保护“文本不裁切”；对偏低严格，对小幅偏高允许飞书导出量化误差。
- 写入失败后必须确认 cleanup 成功，再重新执行 preflight，不能复用半成品页签。
